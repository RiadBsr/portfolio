01

CAMERA PATH — SPIRAL ORBIT DECODE

NEW in V3Core Architecture

Top-Down View — Spiral Path Unwinding From Head Center

WHY SPIRAL: In v2's linear path, objects enter and exit the same direction — they slide in from the right and slide out to the left. This means every exit is visible, requiring careful fade logic. In the spiral, the camera naturally turns away from previous scenes as it orbits to the next. Previous objects end up behind the camera's frustum — they can be culled (geometry disposed) without the user ever seeing the disappearance. This is the same frustum culling principle used in any 3D game engine. The spiral also feels more organic — like unwinding a DNA helix or expanding through a latent manifold.

Spiral Parameters

CAMERA SPLINE DEFINITION

The camera follows a CatmullRom spline through control points arranged on an Archimedean spiral. The spiral starts at radius 2 (units from head center) and expands to radius 45 over the full scroll range. It completes approximately 2.5 revolutions — enough that each scene is 100°+ rotated from the previous one. The camera always looks toward the current scene's focal point (not always toward the head). This means the head itself drops out of frame after S-1, which is fine — the HUD has the name permanently displayed.

CatmullRomCurve3 2.5 revolutions Radius: 2 → 45 LookAt: scene focal
Performance Benefit

NATURAL FRUSTUM CULLING

At any point on the spiral, only the current scene + partial view of the next scene are inside the camera frustum. All previous scenes are behind the camera. In Three.js / R3F, you set frustumCulled: true (default) on scene groups. Objects behind the camera are not rendered — GPU load stays constant regardless of how many scenes exist. For aggressive cleanup, attach a scroll listener that disposes geometry and textures for any scene more than 1 full revolution behind the current camera position.

3D TABLET CHAT UI — IN-WORLD INTERACTION

NEW in V3Killer Feature
The Concept

CHAT HAPPENS ON A 3D TABLET INSIDE THE SCENE

When the user clicks [Chat with Riad] from anywhere (persistent HUD button or S-8 CTA), the following happens:

1. The camera smoothly interpolates to a pre-defined "chat position" in the 3D world — slightly offset from the head, facing a new direction.

2. A low-poly tablet (think iPad-like slab, ~300 faces, beveled edges, matte black) materializes in the void 3 units from the camera. It tilts 8° toward the camera — like someone holding it up for you to see.

3. The tablet's screen is a drei <Html> element rendered on a plane that matches the tablet's screen area. The HTML contains the full chat interface — input field, message history, suggested prompts.

4. 3–4 suggested prompt chips float in 3D space near the tablet (not on the HTML layer — actual 3D text or <Html> chips hovering at ±30° angles from the tablet). They gently bob up and down. Clicking one sends that prompt to the chat.

5. While chatting, the head is visible in the peripheral background, slowly rotating. The particle field continues. The annotation lines are gone. It feels like you're in a research lab, asking the subject questions through a terminal.


Why this is better than a slide-in panel: A slide-in chat panel is what every Intercom/Drift SaaS chatbot does. It's invisible — users have seen it 10,000 times and it creates zero memorable moment. The 3D tablet is a showpiece interaction that no other portfolio has. It demonstrates Three.js + HTML integration mastery. It makes the chat feel like part of the story world, not a bolted-on widget. A recruiter who opens it will remember "the portfolio where the AI chat was on a floating tablet."

Tablet 3D Specifications

THE TABLET OBJECT

Low-poly slab: ~300 faces. Dimensions: 1.8 × 1.2 × 0.06 (Three.js units). Material: MeshStandardMaterial with roughness 0.85, metalness 0.1, color #111. Screen area: a separate mesh plane at the front face, slightly inset (z+0.031). The <Html> component from drei renders the chat interface on this plane. Screen emits a subtle glow (RectAreaLight, intensity 0.15, white).

RoundedBoxGeometry drei <Html> on plane RectAreaLight glow
Transition Animation

CAMERA LERP TO CHAT

When chat is triggered: camera smoothly lerps from current position to chat position over 1.2s (easeInOutCubic). The tablet materializes mid-transition — it scales from 0 → 1 with a slight overshoot (spring physics). Scroll control is disabled during chat mode. Press ESC or click ← to exit — camera lerps back to previous scroll position over 0.8s. Tablet scales down and vanishes.

Suggested Prompts

FLOATING 3D CHIPS

4 prompt chips float in 3D space around the tablet — not on the HTML screen. They are drei <Html> elements positioned at ±30° angles, slightly above and below the tablet plane. They bob gently (sin wave, 0.3 amplitude, 2s period). Clicking a chip sends it as a message and all chips animate away (scale to 0 over 0.3s). Recruiter-optimized prompts: "Tell me about GoPro" / "What are you looking for?" / "Your availability?" / "3D + AI background".

AUTO-DRIFT IDLE BEHAVIOR

NEW in V3
The Problem

"SCROLL TO DECODE" ISN'T ENOUGH

v2 relied on a text prompt: "SCROLL TO DECODE ↓". Problems: (1) On mobile, "scroll" isn't the right word — users swipe. (2) Some users don't read small text. (3) The prompt assumes users know this is a scrollable experience, not a static portfolio page. If someone lands and doesn't scroll within 5 seconds, the most important content (GoPro) is never seen.

The Solution

THE CAMERA TEACHES ITSELF

After 3.5 seconds of no user input (no scroll, no touch, no mouse movement), the camera begins drifting backward along the spiral at a rate of 0.5% scroll progress per second. This is slow enough to feel contemplative, not rushed. The z-counter at bottom-left begins incrementing — subtle but visible. The moment the user scrolls or touches, auto-drift stops and manual control takes over. The user now understands: scrolling controls the camera. The mechanic has been taught without any text instruction.

Implementation: a useEffect that sets an interval after 3.5s of idle. On any wheel/touch/pointermove event, clear the interval and set a flag userHasInteracted = true. After the first interaction, auto-drift never reactivates.

Performance Budget

GPU BUDGET PER FRAME

Target: 16.6ms per frame (60fps). Budget allocation:

Scene geometry: ≤ 6ms
Particle system: ≤ 2ms
Post-processing (none): 0ms
HTML overlay render: ≤ 3ms
Physics/animation: ≤ 2ms
Buffer swap + overhead: ≤ 3.6ms

No post-processing. No bloom, no SSAO, no DOF. The aesthetic is raw wireframe on black — it doesn't need post effects. This is the single biggest performance win: post-processing can eat 40% of the GPU budget on mobile.

TECHNICAL STACK SUMMARY

Implementation
3D Engine

REACT THREE FIBER + DREI

R3F for the 3D scene graph. drei for Html overlays, CatmullRomCurve3 spline, Line primitives, RoundedBox (tablet). @react-three/postprocessing not used — no post effects needed.

Scroll driver: Lenis or native scroll with a custom scroll → spline-t mapping. Avoid ScrollControls (drei) — it's limited for spiral paths. Use a raw wheel/touch listener that increments a normalized t value (0→1) and maps it to a CatmullRom point.

Chat Backend

MISTRAL API

Reuse your Mistral Msg architecture. Next.js API route → Mistral chat completion. System prompt contains: full CV structured as JSON, GoPro project detailed writeup, availability, preferred roles, personality voice. Stream responses for perceived speed.

Deployment

VERCEL + EDGE

Next.js on Vercel. 3D assets in /public with CDN caching. Chat API as edge function for low latency. glTF models: Draco compressed, max 500KB per scene. Total initial bundle target: <2MB (head + landing only). Lazy-load scene models as scroll approaches them.
